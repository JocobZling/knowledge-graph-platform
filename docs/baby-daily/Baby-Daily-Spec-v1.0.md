# Baby Daily Spec v1.0 —— 宝宝成长日报规范（正式版）

## 1. 文档目标

本文档用于规范《宝宝成长日报 v2》的生成流程、内容结构、连续状态管理、辅食规则、海报样式和质量自检要求。

它不是一次性 Prompt，而是一份长期维护的项目规范。后续所有宝宝成长日报、每日确认、辅食进度更新、成长数据更新和海报生成，都应以本文档为唯一标准。

### 1.1 核心目标

- 每天为宝宝生成一份手机竖屏、大字版、适合老人阅读和微信转发的成长日报海报。
- 通过每日 9:00 的文字确认，避免日期、辅食进度、疫苗状态、成长数据等连续状态出错。
- 将宝宝成长数据、辅食进度、疫苗记录、睡眠便便等特殊情况沉淀为可持续追踪的成长档案。
- 用明确的 Error Checklist 记录历史错误，避免重复犯错。

---

## 2. 固定设计原则

### 2.1 增量更新原则

任何新增规则都只能作为增量添加，不得覆盖、删除或简化已确认的日报内容、海报结构、10条建议、辅食规则和自检清单。

### 2.2 用户最新信息优先原则

如果用户当天提供了新信息，例如新的身高体重头围、辅食变化、疫苗、宝宝不喜欢某食材、睡眠便便异常等，必须以用户最新信息为最高优先级。

### 2.3 连续状态优先原则

每天生成日报时，必须继承最近一次确认后的连续状态，而不是重新推断历史。

### 2.4 文字确认与正式海报分离原则

每日 9:00 只发送文字确认消息，不生成图片、确认海报或计划图。

用户回复后，才生成正式《宝宝成长日报 v2》海报。

### 2.5 不确定先确认原则

如果某个状态不确定，例如当前食材第几天、疫苗观察是否结束、某食材是否完成3天观察，应先在确认消息中询问用户，不得擅自编造。

## 3. 宝宝基础档案

### 3.1 固定档案

```yaml
baby:
  birth_date: 2025-11-30
```

### 3.2 月龄与天数计算规则

所有日期、星期、月龄、出生后天数必须基于当天日期和出生日期自动计算。

禁止使用旧日期、示例日期、上一日月龄或上一日出生后天数。

日报中必须展示：

- 当天日期
- 星期
- 宝宝月龄
- 出生日期
- 出生后天数

## 4. 成长数据管理规范

### 4.1 当前成长数据

截至本文档 v1.0，当前最新成长数据为：

```yaml
growth:
  latest_measure_date: 2026-07-01
  height_cm: 69.1
  height_percentile: P64
  weight_kg: 8.55
  weight_percentile: P64
  head_circumference_cm: 45.3
  head_circumference_percentile: P70
```

### 4.2 更新规则

当用户提供新的身高、体重、头围数据时：

- 必须覆盖旧数据。
- 最新测量日期必须同步更新。
- 后续所有确认消息和海报基础数据区都使用新数据。
- 不得回退到旧测量日期或旧数值。

### 4.3 展示规则

正式海报中必须包含基础数据区：

- 最新测量日期
- 身高
- 体重
- 头围
- WHO百分位

成长趋势区必须展示身高、体重、头围的趋势表达或趋势图。

## 5. 辅食管理规范

## 5.1 总体原则

- 奶为主，辅食为辅。
- 当前阶段采用“米粉 + 单一食材”模式。
- 每次只新增一种主要观察食材。
- 新食材通常连续观察3天。
- 不混合多个新食材同时观察。
- 用户最新确认优先于默认计划。

## 5.2 当前辅食连续状态

截至本文档 v1.0，当前连续状态为：

```yaml
feeding:
  stable:
    - 高铁米粉
  completed:
    - 土豆泥
    - 胡萝卜泥
    - 南瓜泥
  tried_not_completed:
    - 西蓝花: 已连续吃2天，未标记为完成3天观察
  current_main_food:
    name: 菠菜
    day: 1
    note: 当前主要观察食材
  tasting:
    - 苹果: 少量体验，不作为新的3天主要观察食材
```

### 5.3 三天观察原则

对于宝宝愿意吃、无明显抗拒、无明显不适的新食材，执行连续3天观察。

观察重点包括：

- 皮疹
- 便便变化
- 胀气
- 哭闹
- 睡眠变化
- 精神状态
- 食欲变化

### 5.4 提前结束观察规则

三天观察不是强制硬吃满3天。

如果宝宝出现以下情况，可以提前结束该食材观察并更换下一种单一食材：

- 明显不喜欢
- 拒食严重
- 吃得很少
- 反复抗拒
- 用户判断不适合继续

提前结束的食材必须在辅食进度卡中如实标记：

- 尝试未继续
- 宝宝不喜欢
- 提前结束

不得误写成“已完成3天观察”。

### 5.5 不适与暂停规则

如果出现以下情况，应暂停新增食材，并建议咨询医生：

- 过敏
- 腹泻
- 明显呕吐
- 持续哭闹
- 睡眠明显异常
- 精神状态异常

### 5.6 疫苗期间辅食规则

疫苗前后几天暂停新增辅食，保持已引入食材稳定。

恢复新增辅食的条件：

- 疫苗观察期结束。
- 用户确认宝宝无异常。

### 5.7 疫苗记录

当前已知疫苗记录：

```yaml
vaccine:
  hand_foot_mouth:
    date: 2026-06-30
    status: completed
    observation_status: ended
```

## 6. 每日 9:00 确认流程

每日 9:00 执行《宝宝成长日报确认》。该步骤必须是文字确认，不得生成图片。

### 6.1 确认消息必须包含

1. 当天日期、星期、宝宝月龄、出生后天数。
2. 当前成长数据：最新测量日期、身高、体重、头围、WHO百分位。
3. 当前辅食进度：已完成食材、当前食材、当前第几天、是否处于疫苗暂停期。
4. 疫苗状态：如有疫苗安排或观察期，明确提示。
5. 特殊记录：睡眠、便便、皮肤、精神状态等。
6. 询问用户今天是否有变化。

### 6.2 用户回复后的处理

如果用户回复“无变化”：

- 沿用当前连续状态。
- 立即生成正式《宝宝成长日报 v2》海报。

如果用户提供更新信息：

- 先更新连续状态。
- 再生成正式《宝宝成长日报 v2》海报。

如果用户只是纠错或补充状态：

- 先更新连续状态。
- 按用户要求决定是否重新生成海报。
- 不得擅自生成确认图。

## 7. 正式海报固定结构

正式海报必须是手机竖屏、大字版，适合老人阅读和微信转发。

### 7.1 固定模块

正式海报必须包含以下模块，不得省略、替换或简化：

1. 标题：《宝宝成长日报 v2》
2. 顶部信息
3. 基础数据区
4. 成长趋势区
5. 今日辅食计划
6. 辅食进度卡
7. 每日10条建议
8. 今日重点总结

### 7.2 顶部信息

必须包含：

- 日期
- 星期
- 月龄
- 出生日期
- 出生后天数
- 今日一句话

### 7.3 基础数据区

必须包含：

- 最新测量日期
- 身高
- 体重
- 头围
- WHO百分位

### 7.4 成长趋势区

必须包含：

- 身高趋势
- 体重趋势
- 头围趋势

### 7.5 今日辅食计划

必须明确：

- 今天的主要食材
- 第几天
- 是否暂停新增辅食
- 是否有少量体验食材
- 观察重点

### 7.6 辅食进度卡

必须展示：

- 已完成观察的食材
- 尝试未继续或提前结束的食材
- 当前观察食材
- 当前第几天
- 下一食材计划或等待条件

### 7.7 今日10条建议

每日必须包含以下10个模块：

1. 发育观察
2. 互动游戏
3. 语言输入
4. 睡眠护理
5. 日常护理
6. 辅食建议
7. 观察记录
8. 奶量与饮水
9. 安全提醒
10. 妈妈小贴士

### 7.8 今日重点总结

以一句或两句清晰总结今日最重要事项，例如：

- 今日重点是继续观察菠菜第1天，苹果仅少量体验。
- 若出现皮疹、腹泻、明显哭闹或睡眠异常，应暂停新增并观察。

## 8. 图片与版式规范

### 8.1 版式要求

- 手机竖屏。
- 大字版。
- 模块化卡片布局。
- 少文字、重点突出。
- 适合老人阅读。
- 适合微信转发。

### 8.2 食材图片要求

食材图必须与文字一致。

特别注意区分：

- 西蓝花
- 菠菜
- 南瓜
- 胡萝卜
- 苹果

如果图像不确定，宁可使用文字卡片或简单图标，不得画错。

### 9.2 状态更新规则

- 用户提供新数据，立即覆盖旧状态。
- 用户纠错时，先修正状态，不急于生成图片。
- 用户说“无变化”，沿用当前状态。
- 用户说某食材宝宝不喜欢，标记提前结束，不得继续强推。

## 10. 异常情况处理规范

### 10.1 疫苗

疫苗日前后几天：

- 暂停新增辅食。
- 维持已引入食材。
- 重点观察体温、精神、睡眠、食欲、皮肤、接种部位。

### 10.2 拒食或不喜欢

如果宝宝明显不喜欢某食材：

- 不强行吃满3天。
- 可提前结束。
- 新食材从第1天重新观察。

### 10.3 疑似过敏或明显不适

如出现皮疹、腹泻、呕吐、持续哭闹、精神差等：

- 暂停新增食材。
- 保持已确认安全食材。
- 建议咨询医生。

## 11. Error Checklist

生成确认消息或正式海报前，必须逐项自检。

### 日期与基础信息

- E001：不得使用旧日期。
- E002：不得使用旧星期。
- E003：不得使用旧月龄。
- E004：不得使用旧出生后天数。
- E005：必须基于出生日期 2025-11-30 自动计算。

### 成长数据

- E006：不得把最新测量日期写错。
- E007：用户更新成长数据后，必须覆盖旧数据。
- E008：不得回退到旧测量日期或旧身高体重头围。

### 辅食进度

- E009：不得把辅食进度重置为第1天。
- E010：不得把已完成观察的食材写成当前观察食材。
- E011：不得把未完成3天的食材写成已完成。
- E012：不得把宝宝不喜欢或提前结束的食材写成已完成3天观察。
- E013：如果用户说明宝宝不喜欢某食材，应优先调整节奏，不得强行推进到第2天、第3天。
- E014：今日辅食计划必须与辅食进度卡一致。
- E015：苹果等少量体验食材不得误记为主要3天观察食材，除非用户明确指定。

### 疫苗

- E016：疫苗日前后几天不得推荐新增辅食。
- E017：必须明确写暂停新增、维持已引入食材。
- E018：恢复新增辅食必须以用户确认宝宝无异常为前提。

### 海报结构

- E019：不得只保留确认流程而丢失日报内容。
- E020：正式海报必须包含全部8个固定模块。
- E021：每日10条建议不得省略。
- E022：不得生成系统计划图代替日报海报。
- E023：不得生成确认海报代替正式日报。

### 9:00确认流程

- E024：9:00确认必须是文字确认，不得生成图片。
- E025：用户回复确认后，必须生成正式《宝宝成长日报 v2》海报。
- E026：用户纠错时，先更新状态，不得擅自生成确认图。

### 图片与食材图标

- E027：食材图标必须与食材名称一致。
- E028：不得把菠菜画成其他蔬菜。
- E029：不得把西蓝花画成其他蔬菜。
- E030：不得把苹果画错。
- E031：图片不确定时，使用文字卡片或简单图标，不得乱画。

### 一致性

- E032：不得在图片不同区域出现互相矛盾的日期。
- E033：不得在图片不同区域出现互相矛盾的月龄。
- E034：不得在图片不同区域出现互相矛盾的辅食天数。
- E035：不得忽略用户当天最新更正。
